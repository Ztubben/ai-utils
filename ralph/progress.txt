## Codebase Patterns
- (Ralph will consolidate reusable learnings here as it works. Read this section before starting each iteration.)
- This is a CLI / tooling repo (bash `ralph.sh`, a pure story-selection engine, a `.ralph.yml` JSON-schema, `bats` shell tests). There is NO frontend — skip browser verification.
- The green gate is the TEST SUITE: fixture-driven unit tests for pure logic (selection engine, config validator) and `bats` tests driving `ralph.sh` against mocked `claude`/`gh` on PATH. Add typecheck/lint only if the project later configures them.
- Design decisions are recorded in `CONTEXT.md` (glossary) and `docs/adr/0001–0005`. Honor that terminology (e.g. HIL, not HITL) and do not contradict an ADR without calling it out.
- META NOTE: this loop (snarktank-style, prd.json-driven) is being used to BUILD a different, issue/label-driven Ralph. Don't confuse the two. The prd.json here is the build backlog; the thing being built reads GitHub Issues.
- TOOLING AVAILABLE: jq, python3.10 (+ jsonschema 4.26, PyYAML), node/npm. NO bats, NO pytest, NO shellcheck. Unit tests use stdlib `unittest`; bats tests will need bats installed (or a mock) when US-011 lands.
- LAYOUT of the tool being built: `bin/ralph` (bash CLI) delegates to `lib/*.py` (pure Python logic), schemas in `schema/`, sample config `.ralph.yml.sample`, tests in `test/` run by `test/run.sh`. See root `AGENTS.md`.
- TEST-FIRST PATTERN: pure logic lives in a Python module returning a result object (ok/errors/resolved), the bash CLI is a thin wrapper that prints + sets exit codes. Unit tests hit the module directly AND a few subprocess tests assert CLI exit codes.
- Run the gate with `./test/run.sh` from repo root.
- Branching: `main` does not exist locally; story branch `ralph/ralph-loop` was created from `master` (which holds CONTEXT.md + ADRs).
- STORY SHAPE: a backlog "story" is a GitHub issue in `gh issue view --json number,title,labels,body` shape (labels are `{"name": ...}` objects). Represent story fixtures as that JSON. `lib/ralph_story.py` normalizes labels (accepts objects OR plain strings) and is the canonical story-FORMAT checker (shape of ONE issue). Keep it distinct from the future selection ENGINE (US-004: ordering/dependency-satisfaction over MANY stories). Story fixtures live under `test/fixtures/stories/{valid,invalid}/`.
- SKILLS the tool ships live in `skills/<name>/SKILL.md`; a skill's `examples/` hold well-formed sample outputs and a unit test asserts every example still validates (drift guard). This is how "verified by example rather than CI" stories still get a green gate.
- Canonical label scheme (ADR-0002): exactly one `state:` (ready|in-progress|awaiting-bench|blocked), one `type:` (afk|hil), one `prio:N`; body has `## Acceptance Criteria` (>=1 `- [ ]`) + `Depends on:` line; HIL adds `## Bench Test Procedure`. A design-decision Blocker carries `ready-for-human` and is kept OUT of `state:ready` (not a third type).
- DEPENDENCY SATISFACTION (ADR-0002): a `Depends on: #N` edge is satisfied only when #N is CLOSED (AFK once merged, HIL once bench-verified — both surface as closed). So any backlog scan must request gh's `state` field. TWO "state"s: gh `state` (OPEN/CLOSED) vs the `state:` LABEL (workflow) — keep them distinct. `lib/ralph_select.py` is the pure selection engine (normalize → select_next → Action; kinds resume/start/no-work/halt); ordering is prio asc then lowest issue# (FIFO). Backlog fixtures = JSON arrays of gh-shaped issues under `test/fixtures/backlogs/`.
- COMPLETION STAGES (side effects: merge/close/PR) use a **plan → run** split so they stay testable: a PURE planner returns the ordered git/gh commands as argv lists (unit-test the plan + its safety guards, e.g. "refuse base==main"), and a `run_plan` runner executes them fail-fast (arg-lists, `shell=False`, combined stdout+stderr, low-verbosity). Integration-test the CLI by writing mock `git`/`gh` scripts that append their argv to `$RALPH_LOG` and prepending their dir to PATH — this IS the "mocked gh/git" gate before bats lands (US-011). `afk_merge` (merge|squash|rebase) maps 1:1 onto `gh pr merge` flags. A completion "closes" the issue; `ralph_select` reads gh CLOSED state to count a dependency satisfied — the stages connect through gh state, not a shared call, so tie an AC like "unblocks dependents" back to select by flipping a fixture's `state` OPEN→CLOSED.
- AGENT PROMPTS vs CODE: judgment-heavy behavior (the TDD iteration itself) lives in a checked-in **prompt** under `prompts/*.md`, NOT in Python. Give each prompt a green gate by drift-guarding its required directives with a unit test (assert key needles present, e.g. red/green/off-target/gating/{issue}/{slug}, and that forbidden terms like "HITL" are ABSENT — note assertNotIn matches the literal substring anywhere, so don't even quote the forbidden term to say "don't use it"). The deterministic, host-testable seams of a stage go in `lib/*.py` behind a thin `bin/ralph` flag (same result-object + CLI-wrapper pattern). `run_gating` shells configured steps in order, fail-fast, low-verbosity (surface only a failing step's output). Gating-config fixtures live under `test/fixtures/gating/` and must be schema-valid (need version+gating+notify).
- FAILURE HANDLING (ADR-0004, `lib/ralph_failure.py`): same plan→run split. A failed **Attempt** is an iteration that ends without green; record it as an issue comment carrying `ATTEMPT_MARKER` (distinct from `HANDOFF_MARKER`). `count_attempts` = count `ATTEMPT_MARKER` comments AFTER `ralph_handoff.non_handoff_comments` filters checkpoints (belt-and-suspenders: markers already differ, but this honors the US-008 seam so a context-full checkpoint is provably NOT an Attempt). `attempt_plan(story, reason, max_attempts)` posts ONE terse comment; when `attempt_no >= max_attempts` it also emits `gh issue edit --add-label state:blocked --remove-label state:<current>` and sets `plan.blocked`. `circuit_breaker_plan(raw_backlog, github_handle, circuit_breaker)` reuses `ralph_select.normalize`, counts OPEN `state:blocked` stories, and at threshold applies `needs-human` to the highest-numbered blocked story + tags `@handle`. TIE-BACK pattern for "loop halts": don't re-implement halt — assert that adding the `needs-human` label flips `ralph_select.next_action` START→HALT (select already treats needs-human anywhere as halt). Config exposes `config["limits"]["max_attempts"|"circuit_breaker"]` and `config["notify"]["github"]` (always present after defaults). Judgment-heavy "fail fast / re-attempt kicked-back state:ready HIL story with a NEW failing test on a fresh PR" lives in `prompts/failure.v1.md` (drift-guarded).
- NOT EVERY STAGE NEEDS plan→run. That split exists to make git/gh SIDE EFFECTS testable. A seam with no side effects — e.g. two-tier memory (US-010, `lib/ralph_memory.py`), which is pure FILESYSTEM QUERIES over `AGENTS.md` — is just plain functions returning data + a thin CLI wrapper (no `Plan`/`run_plan`). Reach for plan→run only when there's something to mutate. When walking dirs toward a `root` that may not be an ancestor of the start, bound the loop by BOTH `cur == root` and `dirname(cur) == cur` so it always terminates.
- MEMORY (ADR-0005, `lib/ralph_memory.py`): there is NO `progress.txt` in the tool being built (the `progress.txt` here in `ralph/` is only the snarktank BUILD HARNESS — keep them separate). Durable Learnings live in nested `AGENTS.md`; story-specific notes live on the issue. `nested_agents_md(start,root)` = read-at-story-start list (nearest-first); `promotion_target(changed_path,root)` = nearest existing `AGENTS.md`, else a NEW one in the changed file's OWN dir (module-local, never the root). Judgment-heavy discipline lives in `prompts/memory.v1.md` (drift-guarded).
- CHECKPOINT/RESUME (ADR-0004, `lib/ralph_handoff.py`): Ralph never compacts — when context fills, an iteration writes a Handoff (WIP commits pushed to the story branch + an issue comment carrying `HANDOFF_MARKER`) and terminates; the next iteration resumes the same state:in-progress story (selection already does resume-first, US-004). Same plan→run split as afk/hil. The Handoff comment marker is the mechanism for "a context-full checkpoint is NOT a failed Attempt": `non_handoff_comments(comments)` filters checkpoints out, and US-009's attempt counter MUST count over that filtered list (not raw comments). Safety guard is stronger than the completions': refuse when EITHER base OR the computed branch is `main`, and never reference base in any command (so `assertNotIn("develop"/base, tokens)` proves "base untouched"). Comments in gh `--json comments` shape are `{"body":...}` dicts listed oldest-first, so `latest_handoff` takes the LAST marked one; helpers also accept plain strings.

## 2026-07-02 - US-004

- `select_next(normalized) -> Action` with kinds resume/start/no-work/halt. Rules: resume-first (any state:in-progress before any state:ready scan), ordering prio ascending then lowest issue number (FIFO), skip state:blocked and ready-for-human Blockers, `needs-human` anywhere → halt.
