# Per-Feature integration branches with completion-time history rewrite

Previously every green story merged straight into the base branch (default `develop`), so parallel Features interleaved their commits and half-finished Features leaked into `develop`. We now integrate per Feature: all of a Feature's stories are worked **directly on one feature branch**, and `develop` receives the Feature only when it is complete. Orphan Stories keep the old flow (own story branch, direct merge to `develop`).

## Decisions

- **One branch, no per-story branches, no per-story PRs.** The feature branch is deterministically named from the PRD issue (`feature/{issue}-{slug}`, same `branch_name` machinery as story branches) and created lazily off `develop` when the Feature's first story starts. A green AFK story is completed by *commit + push + close issue* — no PR, no merge. A green HIL story is completed by *push + record the completion commit SHA on the issue + `state:awaiting-bench`* — no PR. The Feature's only PR is `feature/* → develop`. Rationale: with stories auto-integrating anyway, per-story PRs were pure overhead; this keeps the PR count at one per Feature.
- **Branch invariant + reset-on-block.** The feature branch contains only Passing story work plus at most one in-progress story's WIP at the tip (guaranteed by resume-first selection). When a story is demoted to `state:blocked`, Ralph rewinds the feature branch to the last Passing story boundary and preserves the failed attempt on a pushed rescue branch, naming it — with a description of the failure — in the demotion comment on the issue. Without this, one blocked story's red WIP poisons every sibling that starts after it.
- **Commit-anchored bench verification.** The human verifies a HIL story at its recorded completion commit, in commit order — never at the moving tip — so sibling stories may continue landing on the branch while a HIL story awaits bench without making the bench verdict ambiguous. Bench-pass = close the issue. Bench-fail = comment observed behavior + flip to `state:ready`.
- **Fix forward as fixups; rewrite history only at completion.** Ralph repairs a bench-failed story at the tip as `fixup!` commits of the story's completion commit — never by mid-feature rebase, which would invalidate outstanding bench anchors and force-push under the human's feet. History rewriting is free at feature completion (all stories closed ⇒ all anchors dead), so that is the single rewrite point. Humans may rebase the feature branch at any time; Ralph therefore hard-syncs it from origin at every iteration start.
- **The completion pass** runs at tick time (the final close of a HIL Feature is a human act Ralph cannot observe at story time). For each `prd`-labeled issue with `state:ready` and ≥1 story, all closed: (1) `rebase --autosquash` to collapse fixups into their story commits — on conflict, abort and keep the linear history (cosmetic degradation only); (2) rebase the cleaned branch onto current `develop`; (3) run full gating — the Feature has never been tested against this `develop`; (4) open the Feature PR and merge it **itself** with a merge commit (never squash), then close the PRD. A conflict in (2) or red gate in (3) is a Feature-level blocker: push the branch, comment details on the PRD, label it `ready-for-human`, move on — it does **not** trip the `needs-human` circuit breaker, because independent feature branches don't poison each other.
- **History policy:** `develop` keeps unsquashed per-story commits with one merge commit per Feature; squashing happens only at the human-owned `develop → main` promotion.
- **Freshness on out-of-Feature deps.** When starting a story that depends on an Orphan Story or a PRD (work that landed in `develop` after the feature branch forked), Ralph first merges current `develop` into the feature branch — a merge, not a rebase, so bench anchors survive; the completion-pass rebase linearizes it away later.

## Considered options

- *Per-story branches + PRs into the feature branch*: rejected — PR sprawl was the problem being solved.
- *Freeze a Feature while any story awaits bench*: rejected — commit-anchored verification preserves bench isolation without stalling the Feature.
- *Rebase the fix into the faulty commit immediately on bench-fail*: rejected — invalidates every outstanding bench anchor and races the human's own rebases.
- *Squash-merge the Feature into `develop`*: rejected — per-story history in `develop` is wanted; squashing belongs at `develop → main`.
